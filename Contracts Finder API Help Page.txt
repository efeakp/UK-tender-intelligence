Introduction
This document sets out to describe the mechanisms by which eOpportunity providers and others can feed their
opportunities to the Contracts Finder Service. This can be done via this WebAPI individually at the point of
publication, or via bulk import at a convenient time. Both mechanisms use common interfaces, with the latter
accepting collections of data in one go.
Scope and purpose
The purpose of this document is to describe the interfaces available to approved external notice publication providers
that enable them to publish their notices in the Contracts Finder service.
Notice types that can be uploaded are:
Future Opportunities, i.e. pre-procurement notices to publicise forthcoming activity
Live Opportunities, i.e. invitations to tender
Awards, i.e. transparency publications of awarded contract details
Early Engagements, i.e. notices of intent which invite discussion on the way forward
Whilst this is primarily a specification document to describe the inbound interface, some consideration shall be given
to explain how particular aspects of the specification are handled by the inbound feed, specifically around the
handling of category and region information. It shall describe, where appropriate, any specific data types and field
length limitations placed upon external providers, in order that their opportunities are aggregated within the
Contracts Finder service.
This document shall provide an overview of how warnings and error messages shall be communicated to providers,
but at this time shall not include specific warnings or errors.
Technical Specification
REST
Representational State Transfer (REST) is an architectural style based on the architecture of the Web and embraces
HTTP. Clients initiate requests to servers; servers process requests and return appropriate responses. Requests and
responses are built around the transfer of "representations" of "resources". The central resource with respect to the
Contracts Finder service is the procurement opportunity.
Unique provider identifier
There is a potential for confusion since each notice will have 2 unique identifiers. One is provided by the creator in a
format meaningful to them and must be unique within the creating organisation. Another is assigned internally by
Contracts Finder, which will be a GUID.

Security and Authorisation
Authentication is required to access certain parts of the Web API. To obtain an access token you must POST to the
token endpoint. The POST request must include a basic authorization header containing username:password in
base-64 encoded format. It must also have a body with type application/x-www-form-urlencoded specifying
grant_type=client_credentials. The username and password are the Sid4Gov login credentials.
In response you are granted an access token, a sample response may look like:
{ "access_token":"2YotnFZFEjr1zCsicMWpAA", "expires_in":3600, "token_type":"bearer" }
Headers
To request either JSON or XML MIME types the request needs to contain the appropriate Accept header,
application/json or application/xml respectively.
To access authenticated endpoints a bearer token is needed in the HTTP authorization header, this is the returned
access_token described in section 2.4, for example "bearer 2YotnFZFEjr1zCsicMWpAA".
Error Handling
The standard HTTP status codes are used to indicate errors throughout the system from invalid authentication to
validation errors. Please see http://www.w3.org/Protocols/rfc2616/rfc2616-sec10.html for details on the status codes.
For specific validation errors a 400 status code response is returned with a list of validation errors.
The following is an example:
<ArrayOfValidationError>
<ValidationError>
<Message i:nil="true"/>
<Code>NotFound</Code>
<Property>PublishedNoticeId</Property>
</ValidationError>
</ArrayOfValidationError>
Line Breaks
Some text fields support the use of line breaks to allow long passages to be divided into paragraphs. For example,
the description field in notices.
In order to include a line break you need to insert a CR/LF combination. In JSON this is encoded as "\r\n", and in XML
it is encoded as "&#13;&#10;".

The region(s) of a notice applies to the location of works for the contract - the delivery location. Providers are asked
to supply either one or more regions and/or postcode and/or a region of "Any Region". The provider must supply
one, but choice is dependent on preference and the scope of the delivery.
Postcode information provided shall be used to look-up geo-co-ordinates and a corresponding region.
Email Addresses
Email addresses must conform with the standard pattern, i.e. similar to given.family@domain.ext. There is no
further validation checking.
Country
Country must be one "name" value selected from the countries API method
There is no further validation checking.
Dates and times
Combined date-time values are used, formatted as YYYY-MM-DDThh:mm:ss, e.g. 2014-01-29T15:09:00Z for exactly
3:09pm on 29 January 2014.
The time part is included even for date-only fields.
All times are local UK times, in GMT or BST depending on which is in effect on the date. Timezone information is
appended to the date-time such that “Z” indicates GMT (October to March) and “+01:00” indicates BST (March to
October).
The time part is valid for the DeadlineDate field for notices published from 1 August 2020.
Regions
Region must be a list of one or more comma separated "name" values from the regions API method. Note that API
names have commas removed and accented characters changed to normal ones compared with names shown on
the website. For example, "Saint Helena Ascension and Tristan da Cunha" instead of "Saint Helena, Ascension and
Tristan da Cunha", "Curacao" instead of "Curaçao" and "Saint Barthelemy" instead of "Saint Barthélemy".

This operation is used to upload an individual notice. The payload
consists of a single notice definition, configured to be an Early
Engagement, a Future Opportunity, or a Live Opportunity.
POST Draft/Notice/Copy
This operation will create a copy of an identified notice using the
provided new notice identifier. The notice being copied is identified
by the internally assigned GUID. The provided identifier should be
the user-defined notice id, which cannot be the same as an existing
notice - a new GUID will be assigned and returned to the caller.
POST Draft/Notice/CopyAwarded
This operation will create a copy of an identified awarded notice
using the provided new notice identifier and the new type. The
notice being copied is identified by the internally assigned GUID.
The provided identifier should be the user-defined notice id, which
cannot be the same as an existing notice - a new GUID will be
assigned and returned to the caller.
POST Draft/Notice/Convert
Converts an existing future opportunity into a current opportunity.
POST Draft/Notice/Update
This operation allows an existing notice to be updated. A Notice
object is provided including the notice id and its new state. The
provided details will replace those currently stored.
Version 2 = POST
api/rest/2/publish_draft_notice/{MimeType}
Version 1 = POST Draft/Notice/Publish
This operation allows an existing notice to be published. This will
make the current state of the notice visible to the public through
the Contracts Finder online service. The publish request may
include comments that are recorded against the publish action. If
the VisibleToPublic flag is false, notifications will be suppressed to
anybody watching this notice - this is only appropriate for minor
modifications to existing notices.
POST Draft/Notice/Validate
This operation is used to simply validate the notice. This is used
when validation has already failed after a user has attempted to
publish the notice. Needed because we refresh the page if
validation fails - losing the error messages.
POST Draft/AdditionalDetail/Add
This operation is to be used to upload additional information
relating to a single notice. The payload consists of a single
additional detail definition. The operation requires the identity of a
previously created notice, with which the additional detail data will
be associated. This requires the caller to have access to the GUID
identifying the notice, which could be obtained using
Draft/Notices/Lookup.
